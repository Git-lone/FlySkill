# 需求澄清 Skill

在开始实施任务前，主动识别并澄清模糊或不完整的需求，确保开发方向正确。

## 何时使用

当用户的需求满足以下任一条件时，自动触发此 Skill：

- **缺少关键信息**：需求中缺少技术栈、数据格式、目标指标、业务规则等关键细节
- **使用模糊描述**：包含"优化一下"、"改进性能"、"美化界面"、"加个功能"等不明确的表述
- **多种实现方式**：需求可以有多种理解或实现方式，需要确认具体方向
- **业务逻辑不明确**：需求涉及业务逻辑，但未说明具体规则、流程或判断条件
- **用户体验不明确**：需求涉及用户体验，但未说明目标用户群体、使用场景或交互方式
- **数据处理不明确**：需求涉及数据处理，但未说明数据来源、格式、量级或处理方式
- **集成对接不明确**：需求涉及集成或对接，但未说明第三方系统的接口、协议或数据格式

## 何时不使用

以下情况不应触发此 Skill：

- 用户已经提供了详细的技术规格和实现细节
- 用户明确指出"先按默认方式实现"或"使用标准做法"
- 简单的 bug 修复（有明确的文件名和行号）
- 纯粹的代码重构（不涉及功能变更）
- 用户正在进行探索性对话，还未到实施阶段

## 工作流程

### 第 1 阶段：识别模糊点

1. 分析用户的需求描述
2. 识别缺失的关键信息
3. 判断需求的复杂度和模糊程度

### 第 2 阶段：结构化提问

根据需求类型，智能选择相关的问题类型进行提问。使用勾选框形式，方便用户快速回答。

#### 问题分类体系

**1. 功能需求类**
- [ ] 这个功能的核心目标是什么？要解决什么问题？
- [ ] 目标用户是谁？（内部员工 / 外部客户 / 管理员 / 其他）
- [ ] 是否有类似的参考实现或竞品功能可以参考？
- [ ] 这个功能的优先级如何？是否有上线时间要求？

**2. 技术约束类**
- [ ] 需要使用特定的技术栈或框架吗？
- [ ] 是否需要兼容现有系统或接口？
- [ ] 是否有安全或合规要求？（如数据加密、权限控制）
- [ ] 是否需要考虑向后兼容性？

**3. 数据相关类**
- [ ] 数据来源是什么？（用户输入 / API / 数据库 / 文件上传）
- [ ] 预期的数据格式是什么？（JSON / CSV / XML / 其他）
- [ ] 预期的数据量级是多少？（影响技术选型和性能优化）
- [ ] 数据是否需要持久化？如果需要，存储在哪里？

**4. 用户体验类**
- [ ] 期望的交互方式是什么？（表单 / 弹窗 / 页面跳转 / 其他）
- [ ] 是否有界面风格或设计规范要求？
- [ ] 是否需要支持移动端？
- [ ] 是否需要考虑无障碍访问（Accessibility）？

**5. 性能与规模类**
- [ ] 是否有响应时间要求？（如页面加载 < 2秒）
- [ ] 预期的并发用户数是多少？
- [ ] 是否需要考虑扩展性？（未来可能的功能扩展）
- [ ] 是否需要支持离线使用或缓存？

### 第 3 阶段：处理用户回答

根据用户的回答情况，采取不同的处理策略：

**情况 1：用户完整回答**
- 整理所有回答，生成需求确认文档
- 如果仍有模糊之处，继续追问

**情况 2：用户拒绝回答某些问题**
- 使用行业最佳实践作为默认值
- 在文档中明确标注这些假设

**情况 3：用户回答"随便"或"都可以"**
- 主动推荐一个方案
- 说明推荐理由（基于最佳实践、性能、可维护性等）

**情况 4：用户的回答之间有冲突**
- 指出具体的矛盾之处
- 请求用户澄清优先级或调整需求

**情况 5：需求过于复杂**
- 建议将需求拆分成多个子任务
- 提供拆分方案供用户选择

### 第 4 阶段：生成需求确认文档

创建一份结构化的需求文档，保存到 `docs/requirements-YYYYMMDD.md`，包含以下部分：

1. **需求概述**：一句话总结需求的核心目标
2. **详细需求说明**：按问题类型组织的详细需求
3. **技术方案**：建议的实现方式和技术选型
4. **验收标准**：如何判断需求已完成（可测试的标准）
5. **风险和假设**：明确列出的假设条件和潜在风险
6. **实施计划**：分阶段的实施步骤
7. **未解决的问题**：仍需进一步讨论的事项（如果有）

### 第 5 阶段：确认并开始实施

1. 向用户展示需求确认文档
2. 询问用户是否需要修改
3. 得到确认后，根据文档开始实施任务

## 输出格式

为了确保准确理解你的需求，请回答以下问题：

### 功能需求类
- [ ] 问题 1
- [ ] 问题 2

- [ ] 问题 4

## 注意事项

1. **智能筛选问题**：不要一次性问所有问题，根据需求类型智能选择相关问题
2. **避免过度提问**：如果需求已经比较明确，只问最关键的 3-5 个问题
3. **保持对话自然**：使用友好的语气，避免像填表一样机械
4. **优先级排序**：先问最关键的问题，次要问题可以在后续阶段追问
5. **文档位置**：如果 `docs/` 目录不存在，自动创建
6. **文件命名**：使用当前日期生成文件名，格式为 `requirements-YYYYMMDD.md`
7. **版本管理**：如果同一天有多个需求，在文件名后添加序号（如 `requirements-20241216-02.md`）
