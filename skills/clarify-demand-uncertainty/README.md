# clarify-demand-uncertainty Skill

在实施任务前主动澄清模糊的需求，通过结构化提问确保需求明确，避免返工和误解。

## 📋 概述

这个 Skill 能够：
- ✅ 自动识别模糊或不完整的需求
- ✅ 通过结构化问题澄清关键信息
- ✅ 生成标准化的需求确认文档
- ✅ 智能处理各种特殊情况
- ✅ 避免因需求不明确导致的返工

## 🎯 适用场景

### 会自动触发的情况

- 用户提出的需求缺少关键信息（如技术栈、数据格式、目标指标等）
- 用户使用模糊的描述词（如"优化一下"、"改进性能"、"美化界面"）
- 用户的需求可能有多种实现方式，需要确认具体方向
- 用户提出的需求涉及业务逻辑，但未说明具体规则
- 用户的需求涉及用户体验，但未说明目标用户群体
- 用户的需求涉及数据处理，但未说明数据来源和格式
- 用户的需求涉及集成或对接，但未说明第三方系统的细节

### 不会触发的情况

- 用户已经提供了详细的技术规格和实现细节
- 用户明确指出"先按默认方式实现"或"使用标准做法"
- 简单的 bug 修复（有明确的文件和行号）
- 纯粹的代码重构（不涉及功能变更）
- 用户正在进行探索性对话，还未到实施阶段

## 🔧 工作流程

1. **识别模糊点**：分析用户需求，识别缺失的关键信息
2. **结构化提问**：根据需求类型，智能选择相关问题（勾选框形式）
3. **处理回答**：根据用户回答情况采取不同策略
4. **生成文档**：创建标准化的需求确认文档（`docs/requirements-YYYYMMDD.md`）
5. **确认实施**：得到用户确认后开始实施任务

## 📝 问题分类

Skill 会根据需求类型智能选择以下问题类别：

- **功能需求类**：核心目标、目标用户、参考实现、优先级
- **技术约束类**：技术栈、兼容性、安全要求
- **数据相关类**：数据来源、格式、规模、存储
- **用户体验类**：交互方式、界面设计、设备支持
- **性能与规模类**：响应时间、并发支持、扩展性

## 📄 生成的文档结构

生成的需求文档（`docs/requirements-YYYYMMDD.md`）包含：

1. **需求概述**：一句话总结
2. **详细需求说明**：按问题类型组织
3. **技术方案**：推荐的实现方式
4. **验收标准**：可测试的完成标准
5. **风险和假设**：假设条件和潜在风险
6. **实施计划**：分阶段的实施步骤
7. **未解决的问题**：需要进一步讨论的事项

## 💡 特殊情况处理

- **用户拒绝回答**：使用行业最佳实践作为默认值
- **用户回答模糊**（如"随便"、"都可以"）：主动推荐方案并说明理由
- **回答有冲突**：指出矛盾并请求澄清
- **需求过于复杂**：建议拆分成多个子任务

## 🎓 最佳实践

1. **信任 Skill 的判断**：如果 Skill 提出问题，说明确实有需要澄清的地方
2. **认真回答问题**：详细的回答能得到更准确的实现
3. **使用勾选框**：快速选择适用的选项，提高效率
4. **保存需求文档**：将生成的文档提交到版本控制，作为项目文档
5. **定期回顾**：根据实际使用效果，优化问题模板
